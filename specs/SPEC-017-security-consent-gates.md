# SPEC-017: Security & consent gates

## 1. Motivation
The two REQUIRED items before any real org (BACKLOG launch gates): G1 — BYO LLM keys are plaintext-at-rest (named debt from SPEC-014); G2 — outbound nudges have no opt-out (named from SPEC-013 review; also the unpause condition for the scheduler).

## 2. Requirements
### G1 — encryption at rest for BYO keys
- R1: `api_key` values are encrypted with AES-256-GCM before storage: format `enc:v1:<iv>:<tag>:<ciphertext>` (base64 fields). The key-encryption key comes from env `KEY_ENCRYPTION_KEY` (32 bytes, base64) — in prod a Terraform-generated Secret Manager secret (the envelope boundary); random IV per value; auth tag verified on decrypt
- R2: FAIL CLOSED: storing an `apiKey` without a configured KEK → 400 `encryption_unavailable` (keyless configs, e.g. self-hosted base URLs, remain storable). Decrypt failures at resolution → provider resolution returns null → guided fallback (SPEC-014 R6 semantics)
- R3: Legacy tolerance: values without the `enc:` prefix are read as plaintext (pre-gate rows keep working); every WRITE from now on encrypts. GET masking behavior unchanged
### G2 — notification opt-out
- R4: Per-(user, org) preference, default OPTED IN. Self-only endpoints: GET/PUT /api/orgs/:orgId/notification-prefs {optedOut: boolean} (no target userId exists — preference privacy mirrors gap privacy)
- R5: nudge-dispatch skips opted-out users entirely: nothing sent, nothing logged, count excludes them. INVARIANT-5 note: the count-only response already prevents the org from learning WHO opted out
- R6: In-channel control where the nudges arrive: bridge commands `notifications off` / `notifications on` set the preference for the bound org and confirm in plain language
- R7: Terraform: generated KEK secret wired into the server env allowlist; DEPLOY.md checklist item "unpause scheduler" now reads "G2 shipped — safe to unpause once org id is set"

## 3. Trust invariant check
3: BYO keys are org-trust-boundary material; at-rest encryption + Secret-Manager KEK closes the named hole. 5: opt-out is the individual's information (R4 self-only; R5 count-only). No new org-visible surfaces.

## 4. Contracts
PUT/GET /api/orgs/:orgId/notification-prefs → 200 {optedOut} | 400 | 401 | 404 non-member. Crypto module: `encrypt(plain, kek) -> string`, `decrypt(stored, kek) -> string|null` (null on tamper/failure). Migration 007: bridge_notification_prefs(user_id fk, org_id fk, opted_out bool not null, PK(user_id, org_id)).

## 5. Edge cases & failure modes
Tampered ciphertext → decrypt null → guided fallback, never a 500. KEK rotation: out of scope, documented (re-PUT keys after rotation). Opt-out with no binding: preference persists and applies whenever a binding appears. `notifications` with no argument → usage guidance.

## 6. Acceptance criteria
- AC1 (itest): with harness KEK set, PUT config with apiKey → 200 masked; scripted-LLM flow still works end-to-end (encryption round-trips through resolution)
- AC2 (unit, at-rest proof — recorded delegation, DB inspection is out of itest's black-box scope): stored row has `enc:v1:` prefix and does NOT contain the plaintext key; decrypt round-trips; tampered value decrypts to null; PUT without KEK → 400 encryption_unavailable; legacy plaintext row still resolves
- AC3 (itest): opt-out via API → dispatch delivers nothing to that user and excludes them from `notified`; opt back in → delivered next dispatch (dedup window respected — use a distinct nudge kind or fresh user)
- AC4 (itest): bridge `notifications off` confirms and suppresses delivery; `notifications on` restores
- AC5 (itest): prefs endpoints are self-scoped: default false on first GET; anonymous 401; non-member 404
- AC6 (infra checklist): KEK secret generated by TF, present in server env allowlist; scheduler note updated

## 7. Out of scope
KEK rotation tooling, GCP KMS-managed encryption (Secret-Manager KEK is the demo-tier envelope; KMS is the design-partner upgrade), web settings toggle (lands with the link-code settings UI), digest preferences/quiet hours.

## 8. Decomposition sketch
H01 crypto module + llm.ts integration (unit-red first); H02 prefs migration + endpoints + dispatch filter + bridge command (itest-red first); H03 TF wiring + docs.
